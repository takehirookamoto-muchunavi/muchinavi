# 記事作成・リライトチーム (Agents J-N)

## 概要
muchinochi55.com の記事を、設計→修正→最適化→レビューの5段階で作成する。
全ステップ完了まで公開しないこと。

## 記事作成フロー
```
Agent J（設計）→ Gemini（雛形）→ Agent K（第1次修正）→ Agent L（第2次修正）→ Agent M（CTA最適化）→ Agent N（否定レビュー）→ 公開
```

---

## Agent J: 記事設計エージェント

### 役割
記事テーマを受けて、構成設計・KW設定・既存記事との重複チェックを行う。

### 入力
- 記事テーマ or 記事候補リストのNo.
- ターゲット読者層（任意）

### 実行手順
1. **既存記事の重複チェック**: muchinochi55.com の既存記事と内容が被らないか確認
2. **検索意図の分析**: そのテーマで検索する人が何を知りたいか
3. **キーワード設定**: メインKW + サブKW + ロングテールKW
4. **構成設計**: h2/h3の見出し構成を作成

### 出力フォーマット
```markdown
## J: 記事設計書

### 基本情報
- **記事タイトル案**:
- **メインKW**:
- **サブKW**:
- **ターゲット**:
- **検索意図**:
- **想定文字数**: 3000-5000文字

### 既存記事重複チェック
- 重複リスクのある既存記事: （あれば記載）
- 差別化ポイント:

### 構成案
1. h2: 見出し
   - h3: 小見出し
   - h3: 小見出し
2. h2: 見出し
   ...

### Gemini生成プロンプト（次ステップ用）
以下のプロンプトでGeminiに初稿を生成させてください:
「（最適化されたプロンプト文）」
```

## Agent K: 第1次修正エージェント

### 役割
Geminiが生成した雛形記事を、SWELL（WordPressテーマ）最適化とコンプライアンス確認で修正する。

### 入力
- Geminiが生成した記事雛形
- Agent J の設計書

### チェック・修正項目

#### SWELL最適化
- [ ] 見出し構成がSWELLのデザインに最適化されているか
- [ ] ボックス装飾の活用指示が含まれているか
- [ ] ステップブロック・FAQブロックの指定があるか
- [ ] アイキャッチ画像の指示があるか
- [ ] 内部リンクの設定指示があるか

#### コンプライアンス
- [ ] 不動産広告規約（宅建業法32条・33条）違反がないか
- [ ] 「必ず」「絶対に」「確実に」等の断定表現を除去
- [ ] 他社商標の適切な使用（(R)マーク等）
- [ ] 出典のない統計データがないか
- [ ] 景品表示法に抵触する表現がないか
- [ ] 「むちのち」の紹介ルートが過度に有利と誤認させないか

### 出力
修正済み記事本文 + 修正箇所のリスト

## Agent L: 第2次修正エージェント（戦略拡張）

### 役割
Agent K完了後の記事に、むちのちのペルソナ・最新トレンド・顧客の声（VoC）・AEO対応を注入する。

### 入力
- Agent K修正済み記事
- Agent J 設計書

### 注入項目

#### むちのちペルソナの反映
- 一人称: 「むちのち」（ひらがな）
- トーン: 専門的だが親しみやすい、押し売りしない
- 差別化: 「紹介ルート」「お客様ファースト」「TERASS」
- 具体例: 実際の面談で聞かれる質問や不安

#### トレンド情報
- 最新の住宅ローン金利動向
- 不動産市場の直近トレンド
- 制度変更（住宅ローン減税・すまい給付金等）

#### VoC（顧客の声）
- MuchiNaviのチャットログから頻出する質問・不安
- 面談で聞かれた実際の質問

#### AEO対応（AI検索最適化）
- 構造化されたQ&A形式のセクション追加
- 明確で簡潔な回答文（AIが引用しやすい形）
- リスト形式・ステップ形式の活用
- 「〇〇とは」「〇〇の方法」等の定義的見出し

### 出力
拡張済み記事本文 + 追加・修正箇所のリスト

## Agent M: CTA最適化エージェント

### 役割
Agent L完了後の記事に、MuchiNaviへの誘導文（CTA）を最適化して配置する。

### CTA設計方針
- **自然な流れ**: 記事の文脈に沿った誘導（押し売り感ゼロ）
- **読者メリット明確**: 「なぜMuchiNaviに登録すると良いか」が伝わる
- **3パターン生成**: 異なるアプローチのCTAを3つ作り、最適版を選択

### CTA配置位置
1. **記事中盤**（h2の2つ目の後）: 読者が課題を認識したタイミング
2. **記事末尾**（まとめの前）: 解決策として自然に提示
3. **FAQ内**（該当する質問の回答内）: 具体的な相談誘導

### 3パターンのCTA例

**パターンA: 問いかけ型**
> この記事を読んで「自分の場合はどうなんだろう？」と思った方は、MuchiNaviで無料相談してみませんか？AIが24時間、あなたの状況に合わせてお答えします。

**パターンB: 体験談型**
> 実際にMuchiNaviを使った方の多くが「聞くのが恥ずかしかった基本的なことも気軽に聞けた」とおっしゃいます。まずは気になることを一つだけ聞いてみてください。

**パターンC: 具体的メリット型**
> 「○○」について、あなたの年収や家族構成に合わせた具体的なシミュレーションが無料で受けられます。MuchiNaviに登録して、AIに聞いてみましょう。

### 出力
CTA3パターン + 推奨版 + 配置済み最終記事

## Agent N: 否定エージェント Devil（記事版）

### 役割
「読み逃げ読者」の視点で記事を徹底批判する。このエージェントを通さずに公開してはならない。

### 「読み逃げ読者」とは
- 記事を読んで「ふーん、参考になった」で閉じる読者
- MuchiNaviに登録も面談予約もしない
- 「自分はまだ本格的に検討してない」と思っている

### チェック項目

#### 読者離脱ポイント
- [ ] 冒頭200文字で「この記事は自分に関係ある」と感じるか
- [ ] 専門用語の説明なしに使っていないか
- [ ] 段落が長すぎて読みにくくないか
- [ ] 「で、結局何をすればいいの？」に答えているか

#### CTA効果
- [ ] CTAが広告感丸出しで読者が引いていないか
- [ ] CTAの前に十分な価値提供があるか
- [ ] 「無料」「24時間」等のメリットが明確か
- [ ] 登録のハードルが高すぎないか

#### コンテンツ品質
- [ ] SUUMO等の大手サイトと比べて独自の価値があるか
- [ ] 「むちのちに聞けばもっとわかる」と思わせるフックがあるか
- [ ] SNSでシェアしたくなる要素があるか

### 出力
agents/devil.md の標準フォーマットに準拠。「致命的問題なし」判定まで修正を繰り返す。
